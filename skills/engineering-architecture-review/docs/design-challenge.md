# Design Challenge

Use design challenge mode when the user asks to critique, challenge, validate, pressure-test, or argue against a proposal.

The goal is not to improve the proposal by default. The goal is to decide whether the proposal should exist.

## Rules

- Stop after enough evidence exists to evaluate the proposal.
- Do not attempt repository-wide review.
- Focus only on modules directly involved in the proposed change.
- Prefer decision-oriented output over audit-oriented output.
- Challenge assumptions.
- Compare alternatives.
- Estimate operational cost, migration cost, team cognitive load, and maintenance cost.
- Do not accept proposal framing as fact.

## Challenge

Challenge:

- Assumptions
- Constraints
- Expected benefits
- Migration plans
- Ownership changes
- Operational impact
- Rollback and coexistence
- Long-term maintenance impact

## Output

Use this structure:

1. What is likely correct
2. Assumptions that may be wrong
3. Missing evidence
4. Alternatives considered
5. Cost comparison
6. Risks
7. Recommendation
8. Confidence level

Alternatives must include:

- Current state
- Minimal change option
- Proposed option
